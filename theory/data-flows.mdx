시스톨릭 어레이에서 하드웨어 격자는 동일하더라도, **어떤 값을 PE에 고정하고 어떤 값을 흘려보내느냐**에 따라 세 가지 데이터 흐름 방식이 구분됩니다. WS(Weight Stationary), IS(Input Stationary), OS(Output Stationary)로 불리는 이 방식들은 메모리 접근 횟수, 통신량, 누산 방식에서 서로 다른 효율 특성을 보입니다. SAVANT 프로젝트는 같은 $2 \times 2$ 아두이노 격자 위에서 WS와 OS를 모두 구현하고, 이론적 클럭 수 공식을 통해 두 방식의 효율을 비교합니다.

---

## 1. 데이터 흐름 방식 개요

행렬 곱 $C = A \times B$를 시스톨릭 어레이로 계산할 때, 세 종류의 데이터가 존재합니다.

| 데이터 종류 | 행렬 | 역할 |
|-------------|------|------|
| **Weight** | $B$ (또는 $W$) | 모델 가중치 / 곱해지는 행렬 |
| **Input** | $A$ | 입력 데이터 / 곱하는 행렬 |
| **Output** | $C$ | 누산 결과 |

세 가지 방식은 이 중 **어떤 데이터를 PE 내부에 고정(stationary)** 하느냐로 구분됩니다. 고정된 데이터는 통신 선로를 통해 전송될 필요가 없으므로 통신 오버헤드가 줄어듭니다.

```
┌─────────────────────────────────────────┐
│  WS: Weight 고정  →  Input 흐름         │
│  IS: Input 고정   →  Weight·Partial 흐름│
│  OS: Output 고정  →  A·B 동시 흐름      │
└─────────────────────────────────────────┘
```

## 2. WS (Weight Stationary, 가중치 고정형)

### 원리

WS 방식에서 각 PE는 행렬 $B$(가중치)의 특정 원소를 **펌웨어에 직접 하드코딩**하여 보유합니다. 연산 전에 가중치를 로딩하는 통신이 불필요하며, 계산 단계에서는 **입력 값($A$의 원소)만 격자를 흘러갑니다.**

```
          [A[0][0]] → [PE₀₀: w=B[0][0]] → [PE₀₁: w=B[0][1]]
                              ↓ (부분합)            ↓ (부분합)
          [A[1][0]] → [PE₁₀: w=B[1][0]] → [PE₁₁: w=B[1][1]]
```

- **수평 흐름**: 입력 $A$의 행 원소가 좌 → 우로 이동
- **수직 흐름**: 부분합(Partial Sum)이 위 → 아래로 이동
- **고정값**: 가중치 $B$는 각 PE의 `setup()`에 기록되어 변하지 않음

### SAVANT 구현 (WS)

```cpp
// PE₀₀ 펌웨어 — 가중치 고정형
const int WEIGHT = 3;  // B[0][0] = 3 (컴파일 시 하드코딩)
int partialSum = 0;

void loop() {
  if (clockDetected()) {
    int a_in = receiveFromLeft();     // 왼쪽에서 입력값 수신
    partialSum += a_in * WEIGHT;      // MAC: 가중치는 로컬 상수
    sendRight(a_in);                  // 입력값을 오른쪽 PE로 전달
    sendDown(partialSum);             // 부분합을 아래 PE로 전달
  }
}
```

### WS의 장단점

| 장점 | 단점 |
|------|------|
| 가중치 전송 통신 완전 불필요 | 다른 가중치 행렬에 쓰려면 펌웨어 재업로드 필요 |
| 추론(inference) 시 동일 가중치 반복 적용에 최적 | 가중치 변경 유연성 낮음 |
| SAVANT 통신량이 세 방식 중 가장 적음 | 훈련(training)에는 부적합 |

<Tip>
WS는 신경망 **추론** 시나리오에 최적화되어 있습니다. 학습이 완료된 모델은 가중치가 고정되므로, 수백만 개의 입력에 동일한 가중치를 반복 적용하는 WS가 통신 비용 면에서 가장 효율적입니다. 구글 TPU의 기본 운용 모드도 WS입니다.
</Tip>

## 3. OS (Output Stationary, 출력 고정형)

### 원리

OS 방식에서는 각 PE가 담당하는 **출력 원소 $C[i][j]$의 누산값을 로컬에 유지**합니다. 입력 $A$의 원소는 좌 → 우, 가중치 $B$의 원소는 위 → 아래로 동시에 흐르며, 각 PE는 도착한 $A$와 $B$의 원소를 곱해 로컬 누산기에 더합니다.

```
 A[0][*] →→ [PE₀₀: acc+=A×B] →→ [PE₀₁: acc+=A×B]
              ↑ B[*][0]             ↑ B[*][1]
              ↓                     ↓
 A[1][*] →→ [PE₁₀: acc+=A×B] →→ [PE₁₁: acc+=A×B]
```

모든 연산이 끝난 후 각 PE의 `acc`가 곧 $C$의 해당 원소입니다.

### SAVANT 구현 (OS)

```cpp
// PE(i,j) 펌웨어 — 출력 고정형
int acc = 0;  // C[i][j] 누산기

void loop() {
  if (clockDetected()) {
    int a_in = receiveFromLeft();   // A의 원소 수신
    int b_in = receiveFromTop();    // B의 원소 수신
    acc += a_in * b_in;             // MAC: 로컬 누산
    sendRight(a_in);                // A 원소를 오른쪽으로 전달
    sendDown(b_in);                 // B 원소를 아래로 전달
  }
}

// 연산 완료 후 마스터에게 결과 전송
void reportResult() {
  sendToMaster(acc);
}
```

### OS의 장단점

| 장점 | 단점 |
|------|------|
| 누산이 로컬에서 이루어져 부분합 전송 불필요 | A·B 원소를 모두 전송해야 함 |
| 가중치 유연 변경 가능 | WS 대비 통신량 많음 |
| 훈련·추론 모두에 적용 가능 | |

OS 방식은 SAVANT의 초기 구현 코드에서 먼저 검증되었으며, 이후 WS와의 비교 실험에 활용되었습니다.

## 4. IS (Input Stationary, 입력 고정형)

### 원리

IS 방식에서는 입력값 $A$의 원소가 각 PE에 미리 저장됩니다. 가중치 $B$와 부분합이 격자를 흘러가며 각 PE가 로컬 입력값과 곱셈-누산을 수행합니다.

```
 [PE₀₀: a=A[0][0]] ← B[0][*] 흐름 →→ [PE₀₁: a=A[0][1]]
        ↑ 부분합 흐름                         ↑
 [PE₁₀: a=A[1][0]] ←────────────────→ [PE₁₁: a=A[1][1]]
```

IS는 동일한 입력 배치를 여러 가중치 집합에 반복 적용하는 시나리오(예: 배치 추론에서 단일 입력을 여러 레이어에 통과)에 효율적입니다.

<Note>
IS 방식은 SAVANT 프로젝트에서 현재 구현되어 있지 않습니다. 향후 연구 방향으로, IS 구현 시 WS·OS와의 삼중 비교 실험이 가능합니다. 입력 배치 크기가 변하지 않고 가중치가 자주 바뀌는 시나리오에서 IS가 유리한지를 아두이노 실험으로 검증할 수 있습니다.
</Note>

## 5. 이론적 클럭 수 공식

### Nc 공식

시스톨릭 어레이가 하나의 행렬 곱 연산을 완료하는 데 필요한 **이론적 최소 클럭 수** $N_C$는 다음과 같이 정의됩니다.

$$N_C = 2S_R + S_C + T - 2$$

각 변수의 의미는 다음과 같습니다.

| 기호 | 의미 |
|------|------|
| $S_R$ | 공간 행 차원 (격자의 행 수) |
| $S_C$ | 공간 열 차원 (격자의 열 수) |
| $T$ | 시간 차원 (데이터 스트림 길이) |

### 데이터 흐름별 차원 매핑

행렬 $A$를 $M \times N$, 행렬 $B$를 $N \times P$로 정의하면, 각 방식의 차원 할당은 다음과 같습니다.

| 데이터 흐름 | $S_R$ | $S_C$ | $T$ |
|------------|-------|-------|-----|
| **WS** | $M$ | $N$ | $P$ |
| **IS** | $N$ | $P$ | $M$ |
| **OS** | $M$ | $P$ | $N$ |

### 예시 계산: M=2, N=2, P=4

$2 \times 2$ 행렬 $A$와 $2 \times 4$ 행렬 $B$의 곱에서 각 방식의 클럭 수를 계산합니다.

**WS** ($S_R = M = 2$, $S_C = N = 2$, $T = P = 4$):
$$N_C^{\text{WS}} = 2(2) + 2 + 4 - 2 = 4 + 2 + 4 - 2 = \mathbf{8}$$

**IS** ($S_R = N = 2$, $S_C = P = 4$, $T = M = 2$):
$$N_C^{\text{IS}} = 2(2) + 4 + 2 - 2 = 4 + 4 + 2 - 2 = \mathbf{8}$$

**OS** ($S_R = M = 2$, $S_C = P = 4$, $T = N = 2$):
$$N_C^{\text{OS}} = 2(2) + 4 + 2 - 2 = 4 + 4 + 2 - 2 = \mathbf{8}$$

<Note>
이 예시($M = N = 2$, $P = 4$)에서는 세 방식 모두 $N_C = 8$로 동일합니다. 그러나 **SAVANT의 $2 \times 2$ 격자**에서 OS로 위 예시를 처리하려면 $S_C = P = 4$가 격자 열 수 2를 초과하므로 **타일링(tiling)** 이 필요합니다. 통신 오버헤드와 타이밍 스큐를 감안한 **실제 클럭 수**는 이론값보다 항상 큽니다.
</Note>

## 6. 격자 크기 제약과 타일링

### 격자 크기 제약

SAVANT의 $2 \times 2$ 격자는 $S_R \leq 2$, $S_C \leq 2$로 제한됩니다. 행렬 차원이 이 경계를 넘어설 경우 한 번의 패스(pass)로 전체 곱셈을 완료할 수 없습니다.

### 타일링(Tiling) 원리

타일링은 큰 행렬을 격자 크기에 맞는 작은 블록(타일)으로 나누어 순차적으로 처리하는 기법입니다.

```
16×16 행렬을 2×2 격자로 처리:

 [블록₀₀][블록₀₁]···[블록₀₇]
 [블록₁₀][블록₁₁]···[블록₁₇]
       ·
 [블록₇₀][블록₇₁]···[블록₇₇]

총 8×8 = 64개의 출력 블록 → 각 블록마다 N/S_C = 16/2 = 8 라운드
→ 64 × 8 = 512회 패스 (vs 4×4 격자: 16 × 4 = 64회)
```

### 격자 크기별 타일링 비교 ($16 \times 16$ 행렬)

$$\text{총 패스 수} = \left(\frac{M}{S_R}\right) \times \left(\frac{P}{S_C}\right) \times \left(\frac{N}{T_{\text{block}}}\right)$$

| 격자 크기 | 출력 블록 수 | 라운드/블록 | 총 패스 수 | 비율 |
|-----------|-------------|------------|-----------|------|
| $2 \times 2$ | $8 \times 8 = 64$ | 8 | **512** | 기준 (×16) |
| $4 \times 4$ | $4 \times 4 = 16$ | 4 | **64** | ×1 |
| $8 \times 8$ | $2 \times 2 = 4$ | 2 | **8** | ×1/8 |
| $16 \times 16$ | $1 \times 1 = 1$ | 1 | **1** | ×1/64 |

같은 $16 \times 16$ 행렬에서 $4 \times 4$ 격자는 $2 \times 2$ 대비 **16배 적은 패스**로 연산을 완료합니다. 이 수치는 격자 규모 확장의 효율을 정량적으로 보여 줍니다.

### SAVANT에서의 타일링 구현 방향

현재 SAVANT는 $2 \times 2$ 행렬 곱($M = N = P = 2$)에 대해 타일링 없이 단일 패스로 동작합니다. 향후 실험에서 $4 \times 4$ 이상의 행렬을 다루려면 마스터 보드에서 타일 스케줄러를 구현하고, 각 타일 결과를 모아 최종 $C$ 행렬을 조립하는 과정이 필요합니다.

```cpp
// 마스터 보드 타일 스케줄러 (의사 코드)
void tiledMatMul(int A[][], int B[][], int C[][], int dim) {
  for (int ti = 0; ti < dim/GRID_ROWS; ti++) {
    for (int tj = 0; tj < dim/GRID_COLS; tj++) {
      for (int tk = 0; tk < dim/TILE_K; tk++) {
        // 타일 A[ti][tk], B[tk][tj]를 격자로 전송
        sendTileToGrid(A, B, ti, tj, tk);
        // 격자 연산 완료 대기
        waitForGridDone();
        // 부분 결과를 C[ti][tj]에 누산
        accumulateTile(C, ti, tj);
      }
    }
  }
}
```

<Warning>
$2 \times 2$ 격자에서 $16 \times 16$ 행렬을 타일링으로 처리하면 **512번의 보드-보드 통신 라운드**가 필요합니다. 각 라운드의 통신 오버헤드(~수 ms)가 누적되면 오히려 단일 아두이노 순차 처리보다 느려질 수 있습니다. SAVANT의 성능 이점은 소규모($2 \times 2$) 행렬 실험에서 알고리즘적 구조 차이를 확인하는 데 있으며, 대규모 행렬에서의 실질적 속도 우위는 칩 수준의 통신 지연 없는 하드웨어에서만 실현됩니다.
</Warning>

### 세 가지 데이터 흐름 방식 최종 요약

| 항목 | WS | IS | OS |
|------|----|----|-----|
| 고정 데이터 | 가중치 $B$ | 입력 $A$ | 출력 $C$ (누산) |
| 흐르는 데이터 | $A$, 부분합 | $B$, 부분합 | $A$, $B$ |
| 통신량 | 가장 적음 | 중간 | 중간 |
| 최적 시나리오 | 반복 추론 | 배치 입력 재사용 | 유연한 범용 연산 |
| SAVANT 구현 여부 | ✅ 구현 | ❌ 미구현 | ✅ 구현 |
