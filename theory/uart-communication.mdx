SAVANT 시스톨릭 어레이에서 PE(Processing Element) 역할을 하는 아두이노 보드들은 서로 데이터를 주고받아야 합니다. 보드 내부 SRAM에서 데이터를 읽는 것은 나노초 수준으로 빠르지만, 보드와 보드 사이의 통신은 훨씬 느립니다. 이 페이지에서는 아두이노 보드 간 통신에 사용되는 UART의 원리부터 SAVANT가 채택한 커스텀 클럭 동기 방식까지, 통신 기술 전반을 설명합니다.

## 1. UART란

**UART(Universal Asynchronous Receiver/Transmitter)** 는 두 장치 사이에서 데이터를 직렬(Serial)로 전송하는 통신 규격입니다. 이름에 들어있는 **Asynchronous(비동기)** 는 두 장치가 별도의 공유 클럭 신호 없이 통신한다는 의미입니다.

대신 통신 전에 양측이 **보레이트(Baud Rate, 초당 비트 전송 수)** 를 미리 약속합니다. 약속된 속도에 맞춰 각자의 내부 클럭으로 비트를 읽고 씁니다.

UART는 방향마다 별도의 선을 사용합니다.

| 신호선 | 역할 |
|-------|------|
| **TX** (Transmit) | 데이터 송신 |
| **RX** (Receive) | 데이터 수신 |

<Note>
장치 A의 TX와 장치 B의 RX를 서로 연결하고, 장치 A의 RX와 장치 B의 TX를 연결합니다. 한쪽이 보내는 선(TX)을 상대방이 받는 선(RX)에 연결하는 교차 연결 방식입니다.
</Note>

## 2. UART 프레임 구조

UART는 1바이트(8비트)를 전송할 때 다음과 같은 **프레임** 구조를 사용합니다.

```
[유휴 상태: HIGH] → [시작 비트: LOW] → [D0 D1 D2 D3 D4 D5 D6 D7] → [정지 비트: HIGH]
```

| 구간 | 전압 | 역할 |
|------|------|------|
| 유휴(Idle) 상태 | HIGH (5V) | 전송 없음, 대기 중 |
| 시작(Start) 비트 | LOW (0V) | "데이터 전송 시작" 신호 |
| 데이터 비트 (8개) | HIGH/LOW | 실제 데이터 |
| 정지(Stop) 비트 | HIGH (5V) | "전송 완료" 신호, 유휴 상태 복귀 |

총 10비트로 1바이트를 전송합니다. **9600 보레이트**에서의 시간을 계산하면 다음과 같습니다.

- 1비트 전송 시간: 1 ÷ 9600 ≈ **104 μs**
- 1바이트(10비트) 전송 시간: 104 × 10 ≈ **1,040 μs (약 1ms)**

## 3. 보레이트 (Baud Rate)

보레이트는 1초에 전송할 수 있는 비트 수입니다. 송신 측과 수신 측의 보레이트가 **반드시 일치**해야 합니다. 양측의 보레이트가 다르면 수신 측이 비트를 엉뚱한 타이밍에 읽게 되어 데이터가 깨집니다.

| 보레이트 | 1바이트 전송 시간 | 비고 |
|---------|----------------|------|
| 9600 | ~1,040 μs | SAVANT 초기 구현 |
| 19200 | ~520 μs | — |
| 38400 | ~260 μs | SoftwareSerial 안정 상한선 |
| 115200 | ~87 μs | 하드웨어 시리얼 권장 |

<Warning>
SAVANT에서 9600 보레이트로 1바이트를 전송하는 데 약 **1ms**가 소요됩니다. 이는 아두이노 내부 SRAM 접근 속도(나노초 수준)보다 **약 백만 배** 느립니다. 이 통신 오버헤드는 실험 결과 해석에서 핵심적인 요소입니다.
</Warning>

## 4. 하드웨어 시리얼 vs SoftwareSerial

아두이노에서 시리얼 통신을 구현하는 방법은 두 가지입니다.

### 하드웨어 시리얼

ATmega328P 칩 안에 시리얼 통신을 위한 **전용 하드웨어 회로(USART)**가 내장되어 있습니다. D0(RX), D1(TX) 핀이 이 회로에 연결되어 있습니다.

- CPU가 직접 개입하지 않아도 하드웨어가 자율적으로 비트를 처리
- 빠르고 안정적이며 115200 보레이트에서도 오류가 없음
- **단점:** Arduino Nano에는 1쌍(RX/TX)만 존재

### SoftwareSerial

소프트웨어로 UART를 흉내 내는 방식입니다. 임의의 디지털 핀을 RX/TX로 사용할 수 있습니다.

```cpp
#include <SoftwareSerial.h>
SoftwareSerial mySerial(4, 5);  // RX=D4, TX=D5

void setup() {
  mySerial.begin(9600);
}

void loop() {
  mySerial.println("Hello");
}
```

- 하나의 보드에서 여러 시리얼 연결을 만들 수 있음
- **단점:** CPU가 타이밍을 직접 계산하며 핀을 토글하므로 속도와 안정성이 낮음
- 높은 보레이트(38400 이상)에서 타이밍 오차 증가
- 여러 포트를 동시에 수신할 수 없음

<Note>
SAVANT의 중간 보드(PE(0,1), PE(1,0))는 데이터를 받으면서 동시에 다른 방향으로 전달해야 합니다. 하드웨어 시리얼 포트가 1개뿐이므로, SoftwareSerial만으로는 이 요구사항을 충족하기 어렵습니다.
</Note>

## 5. println() vs write() — 통신 최적화

`Serial.println()`과 `Serial.write()`는 모두 데이터를 전송하지만, 전송하는 바이트 수가 다릅니다.

| 함수 | value = 2 일 때 전송 내용 | 전송 바이트 수 | 9600 보레이트 소요 시간 |
|------|--------------------------|--------------|----------------------|
| `Serial.println(2)` | `'2'` + `'\r'` + `'\n'` (ASCII 변환) | 3바이트 | ~3,120 μs |
| `Serial.write(2)` | `0x02` (원시 바이너리) | 1바이트 | ~1,040 μs |

`println()`은 숫자를 사람이 읽을 수 있는 ASCII 문자열로 변환하여 전송합니다. 반면 `write()`는 숫자 값 자체를 1바이트 이진 데이터로 바로 전송합니다.

```cpp
// 비효율적인 방법 (디버깅용으로는 유용)
Serial.println(result);  // 숫자 → 문자열 변환 후 전송

// 효율적인 방법 (SAVANT 최적화 버전)
Serial.write(result);    // 1바이트 그대로 전송
```

<Tip>
보레이트를 9600 → 38400으로 높이고 `println()` → `write()`로 변경하면 통신 속도가 약 **12배** 향상됩니다. 실험에서 통신 오버헤드를 줄이는 간단하고 효과적인 방법입니다.
</Tip>

## 6. 커스텀 클럭 동기 방식 (SAVANT에서 사용)

SoftwareSerial은 보레이트 한계와 동시 수신 불가 문제가 있습니다. SAVANT는 이를 해결하기 위해 별도의 **CLOCK 신호선과 DATA 신호선**을 사용하는 커스텀 동기 통신 방식을 채택합니다.

### 동작 원리

1. 송신 측이 DATA 핀에 비트 값을 설정합니다.
2. 송신 측이 CLOCK 핀을 HIGH → LOW로 펄스(pulse)합니다.
3. 수신 측은 CLOCK 핀의 LOW 신호를 감지하고, 그 시점의 DATA 핀 값을 읽습니다.
4. 위 과정을 8번 반복하여 1바이트를 수신합니다.

### 구현 코드

```cpp
// 송신 측: Arduino 내장 함수 활용
shiftOut(DATA_PIN, CLOCK_PIN, MSBFIRST, value);
// MSBFIRST: 최상위 비트(MSB)부터 전송
```

```cpp
// 수신 측: 커스텀 수신 함수
byte receiveByte(byte clockPin, byte dataPin) {
  byte value = 0;
  for (int i = 0; i < 8; i++) {
    // 현재 DATA 핀 값을 value에 누적
    value = (value << 1) | digitalRead(dataPin);
    // 다음 CLOCK 펄스 대기
    while (digitalRead(clockPin) == HIGH);  // LOW로 떨어질 때까지 대기
    while (digitalRead(clockPin) == LOW);   // HIGH로 올라올 때까지 대기
  }
  return value;
}
```

### 속도 비교

| 통신 방식 | 1바이트 전송 시간 |
|----------|----------------|
| SoftwareSerial 9600 baud | ~1,040 μs |
| SoftwareSerial 38400 baud | ~260 μs |
| 커스텀 클럭 동기 방식 | ~100 ~ 200 μs |

커스텀 클럭 방식의 속도는 `digitalWrite()` / `digitalRead()` 함수 호출 속도(수 μs)에 의해 제한됩니다.

<Note>
공식 SPI 라이브러리를 사용하지 않는 이유: Arduino Nano에는 하드웨어 SPI가 1세트뿐입니다. SPI는 마스터(Master)와 슬레이브(Slave) 역할이 구분되는데, PE(0,1)과 PE(1,0) 같은 중간 보드는 데이터를 받는 동시에 다음 보드로 전달해야 합니다. 즉 마스터와 슬레이브 역할을 동시에 수행해야 하므로 표준 SPI 라이브러리를 그대로 적용할 수 없습니다.
</Note>

## 7. 단방향 통신

시스톨릭 어레이의 중요한 특성 중 하나는 **데이터가 고정된 방향으로만 흐른다**는 점입니다. A 행렬의 데이터는 오른쪽으로, B 행렬의 데이터는 아래쪽으로만 이동합니다.

이 단방향 흐름 덕분에 각 보드 간 연결에 신호선이 **단 1개**만 필요합니다 (양방향 UART처럼 TX와 RX 2개가 필요하지 않습니다). 커스텀 클럭 방식에서는 CLOCK 1개 + DATA 1개, 총 2개 선으로 단방향 전송을 구현합니다.

```
[PE(0,0)] ──DATA→── [PE(0,1)]
    │                    │
   DATA↓               DATA↓
    │                    │
[PE(1,0)] ──DATA→── [PE(1,1)]
```

## 8. 통신 오버헤드와 실험 결과 해석

SAVANT 실험에서 시스톨릭 어레이(실험군)가 폰 노이만 방식(대조군)보다 **느리게 측정될 수 있습니다.** 이는 실험이 잘못된 것이 아니라, 오히려 중요한 공학적 교훈을 담고 있습니다.

| 연산 | 소요 시간 |
|------|----------|
| Arduino 내부 SRAM 접근 | 수 나노초 (ns) |
| SoftwareSerial 9600baud 1바이트 전송 | ~1,040,000 ns (1ms) |
| 커스텀 클럭 방식 1바이트 전송 | ~100,000 ~ 200,000 ns |

보드와 보드 사이의 통신 시간은 내부 메모리 접근보다 **수만 ~ 백만 배** 느립니다. 실제 Google TPU(Tensor Processing Unit) 같은 상용 시스톨릭 어레이는 모든 PE를 하나의 칩 안에 집적하여, 보드 간 통신 없이 나노초 수준의 내부 배선으로 데이터를 전달합니다.

<Note>
**핵심 결론:** SAVANT 실험에서 통신 오버헤드가 크게 나타나는 이유 자체가, 실제 TPU가 왜 PE들을 하나의 칩에 집적하는지를 설명해 줍니다. 시스톨릭 어레이 구조의 진정한 장점은 물리적으로 인접한 PE들 사이의 초고속 내부 통신에서 비롯됩니다. 아두이노 보드 실험은 그 구조적 이점을 하드웨어 제약으로 인해 수치로 보여주기 어렵지만, 동작 원리 자체는 동일합니다.
</Note>
