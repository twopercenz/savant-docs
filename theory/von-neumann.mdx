현대 컴퓨터의 설계 원칙은 1945년 존 폰 노이만(John von Neumann)이 제안한 구조에서 출발합니다. CPU와 메모리가 하나의 공유 버스로 연결되어 명령어와 데이터를 같은 메모리 공간에 저장하는 이 구조는 오늘날에도 대부분의 범용 컴퓨터에 적용되어 있습니다. SAVANT 프로젝트는 이 구조가 행렬 연산에서 드러내는 순차적 메모리 접근 패턴—이른바 **폰 노이만 병목(Von Neumann Bottleneck)**—을 아두이노 대조군 실험으로 모사하고, 시스톨릭 어레이를 통해 이를 완화할 수 있음을 보입니다.

## 1. 폰 노이만 구조 개요

폰 노이만 구조(Von Neumann Architecture)는 다음 네 가지 핵심 구성 요소로 이루어집니다.

| 구성 요소 | 역할 |
|-----------|------|
| **CPU** | 연산 수행 (ALU) 및 제어 (Control Unit) |
| **메모리** | 명령어와 데이터를 동일한 주소 공간에 저장 |
| **입출력 장치** | 외부와의 데이터 교환 |
| **시스템 버스** | CPU ↔ 메모리 ↔ I/O를 연결하는 단일 통로 |

이 구조의 결정적 특징은 **명령어와 데이터가 같은 메모리에, 같은 버스를 통해 오간다**는 점입니다. CPU는 프로그램을 실행하기 위해 아래의 사이클을 반복합니다.

### 페치-디코드-실행(Fetch–Decode–Execute) 사이클

```
┌─────────────┐     버스      ┌─────────────────────┐
│   Memory    │◄─────────────►│        CPU          │
│  (명령어 +   │               │  ┌───────────────┐  │
│   데이터)    │               │  │  1. Fetch     │  │  ← 메모리에서 명령어 가져오기
└─────────────┘               │  │  2. Decode    │  │  ← 명령어 해석
                               │  │  3. Execute   │  │  ← 연산 수행 (데이터 접근 포함)
                               │  └───────────────┘  │
                               └─────────────────────┘
```

1. **Fetch**: 프로그램 카운터(PC)가 가리키는 주소에서 명령어를 버스를 통해 CPU로 불러옵니다.
2. **Decode**: CPU 내부의 제어 장치가 명령어를 해석하여 어떤 연산을 수행할지 결정합니다.
3. **Execute**: ALU가 연산을 수행합니다. 피연산자가 메모리에 있다면 다시 버스를 거쳐 데이터를 읽거나 씁니다.

이 세 단계를 거칠 때마다 CPU는 반드시 버스를 통해 메모리에 접근해야 합니다. 연산이 복잡해질수록 버스 사용 횟수는 기하급수적으로 증가합니다.

## 2. 폰 노이만 병목 현상

### 병목의 정의

1977년 ACM 튜링상 수상 강연에서 존 배커스(John Backus)는 이 문제를 공식적으로 **"폰 노이만 병목(Von Neumann Bottleneck)"**이라 명명했습니다. 핵심 주장은 간단합니다.

> "The von Neumann bottleneck … is not a problem of hardware speed, but of the narrow channel through which data must flow."
> — John Backus, 1977

CPU의 연산 속도는 수십 년간 무어의 법칙에 따라 급속도로 향상되었지만, CPU와 메모리를 잇는 **버스의 대역폭(bandwidth)** 은 그 속도를 따라잡지 못했습니다. 이 격차를 **메모리 장벽(Memory Wall)** 이라 부릅니다.

```
CPU 성능 향상 속도  ████████████████████████  (지수 성장)
메모리 대역폭 향상  ████████                  (완만한 성장)
                   ───────────────────────→ 시간
                         ↑ Memory Wall
```

### 행렬 곱셈에서의 병목

$M \times N$ 행렬 $A$와 $N \times P$ 행렬 $B$의 곱을 계산할 때, 폰 노이만 방식 순차 프로세서에서 메모리 접근 횟수는 다음과 같이 증가합니다.

$$\text{메모리 접근 횟수} \approx M \times N \times P$$

예를 들어 $M = N = P = 16$이면 $16^3 = 4{,}096$회의 메모리 접근이 필요합니다. 행렬 크기가 커질수록 접근 횟수는 **세제곱(cubic)** 으로 폭발적으로 증가합니다.

<Note>
AI·딥러닝 워크로드는 수백~수천 차원의 행렬 곱셈을 수천만 번 반복합니다. 폰 노이만 방식의 단일 프로세서로는 이 규모의 연산을 현실적인 시간 안에 처리하기 어렵습니다. 이것이 GPU, TPU와 같은 병렬·전용 하드웨어가 등장한 근본 이유입니다.
</Note>

## 3. 하버드 구조와 아두이노

### 하버드 구조 개요

하버드 구조(Harvard Architecture)는 폰 노이만 구조와 달리 **명령어 메모리와 데이터 메모리를 물리적으로 분리**하고, 각각 전용 버스를 사용합니다.

| 비교 항목 | 폰 노이만 | 하버드 |
|-----------|-----------|--------|
| 메모리 공간 | 명령어·데이터 통합 | 명령어·데이터 분리 |
| 버스 | 공유 단일 버스 | 독립 이중 버스 |
| 동시 접근 | 불가 | 명령어 + 데이터 동시 가능 |
| 복잡도 | 낮음 | 높음 |

### 아두이노 나노(ATmega328P)의 실제 구조

아두이노 나노에 탑재된 ATmega328P는 **수정 하버드 구조(Modified Harvard Architecture)** 를 사용합니다.

- **플래시 메모리(32 KB)**: 컴파일된 프로그램 코드(명령어) 저장
- **SRAM(2 KB)**: 변수, 배열 등 실행 중 데이터 저장
- **EEPROM(1 KB)**: 비휘발성 데이터 저장

두 메모리는 별도의 버스를 통해 CPU에 연결됩니다. 즉, **ATmega328P는 순수한 폰 노이만 구조가 아닙니다.**

<Warning>
**실험 한계: 아두이노는 폰 노이만 병목을 하드웨어적으로 재현하지 않습니다.**

ATmega328P의 하버드 구조에서는 명령어와 데이터가 분리된 버스를 사용하므로, 폰 노이만 병목의 **물리적 원인**(단일 버스 경합)이 하드웨어 수준에서 존재하지 않습니다. 이 점을 보고서에서 명시적으로 밝혀야 합니다.
</Warning>

## 4. 실험에서의 폰 노이만 모사

### 무엇을 모사하는가

SAVANT 대조군 실험에서 아두이노가 재현하는 것은 폰 노이만 병목의 **하드웨어 메커니즘이 아니라 행동 패턴(behavioral pattern)** 입니다.

> **"단일 프로세서가 공유 배열에 순차적으로 접근하는 패턴"**

실제 폰 노이만 구조의 병목은 CPU-메모리 버스 경합에서 비롯되지만, 우리가 측정하고 비교하는 것은 **메모리(배열) 접근 횟수**와 **총 연산 시간**입니다. 이 두 지표에서 순차 처리와 병렬 처리의 차이를 정량적으로 드러낼 수 있습니다.

### 구현 방식

대조군 아두이노는 다음과 같은 3중 for-루프로 $2 \times 2$ 행렬 곱을 수행합니다.

```cpp
// 대조군: 순차 행렬 곱셈 (Von Neumann 패턴 모사)
int A[2][2] = {{1, 2}, {3, 4}};
int B[2][2] = {{5, 6}, {7, 8}};
int C[2][2] = {{0, 0}, {0, 0}};

long accessCount = 0;  // 메모리 접근 카운터

void computeSequential() {
  for (int i = 0; i < 2; i++) {
    for (int j = 0; j < 2; j++) {
      for (int k = 0; k < 2; k++) {
        C[i][j] += A[i][k] * B[k][j];  // A와 B 각각 1회 읽기
        accessCount += 2;               // 메모리 접근 횟수 계산
      }
    }
  }
}
```

`accessCount` 변수는 배열 읽기 연산 횟수를 누적합니다. $2 \times 2 \times 2$ 행렬의 경우 총 **8회**의 읽기 접근이 발생합니다($M \times N \times P = 2 \times 2 \times 2$).

### 측정 지표

| 측정 항목 | 측정 방법 | 의미 |
|-----------|-----------|------|
| 메모리 접근 횟수 | `accessCount` 카운터 | 순차 접근의 규모 |
| 총 실행 시간 | `micros()` 타이머 | 절대적 처리 시간 |
| 시간 대비 접근 비율 | 접근 횟수 ÷ 시간 | 접근 효율 |

<Tip>
행렬 크기를 $4 \times 4$, $8 \times 8$, $16 \times 16$으로 키우면서 실험하면 `accessCount`가 세제곱 증가하는 패턴을 직접 확인할 수 있습니다. 이 수치를 실험군(시스톨릭 어레이)의 통신 횟수와 비교하면 데이터 재사용 효과가 명확하게 드러납니다.
</Tip>

### 실험의 의의와 한계 요약

| 항목 | 내용 |
|------|------|
| **모사하는 것** | 단일 프로세서의 순차적 배열 접근 패턴 |
| **모사하지 못하는 것** | 단일 버스 경합이라는 하드웨어적 병목 원인 |
| **측정 가능한 것** | 접근 횟수, 실행 시간, 스케일링 특성 |
| **비교 기준** | 시스톨릭 어레이의 데이터 재사용 횟수 및 병렬 실행 시간 |

이 구분을 명확히 함으로써 SAVANT는 하드웨어 한계를 인정하면서도, 순차 처리와 병렬 처리의 **알고리즘적 효율 차이**를 유효하게 비교합니다.
