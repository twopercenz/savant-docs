1978년 카네기 멜론 대학(CMU)의 H.T. 쿵(H.T. Kung)과 찰스 레이저슨(Charles Leiserson)이 제안한 **시스톨릭 어레이(Systolic Array)** 는 폰 노이만 병목의 직접적인 해법으로 설계된 병렬 컴퓨팅 구조입니다. 데이터가 심장 박동처럼 클럭마다 PE에서 PE로 흘러가면서 연산이 수행되는 이 구조는, 오늘날 구글 TPU의 핵심 엔진으로 채택될 만큼 행렬 연산에서 탁월한 효율을 보입니다. SAVANT 프로젝트는 이 원리를 4개의 아두이노 나노로 구성한 $2 \times 2$ 격자로 구현하여, 시스톨릭 처리가 가져오는 효율 향상을 실험적으로 검증합니다.

## 1. 시스톨릭 어레이 등장 배경

### 폰 노이만 병목에 대한 해답

H.T. 쿵과 레이저슨은 폰 노이만 병목의 핵심이 **CPU와 메모리 사이의 반복적인 데이터 왕복**에 있다고 진단했습니다. 그들의 해법은 단순하면서도 강력했습니다.

> "데이터를 메모리에서 여러 번 꺼낼 것이 아니라, 한 번 꺼낸 데이터를 여러 연산 장치 사이에서 재사용하라."

이를 구현하기 위해 **다수의 처리 요소(Processing Element, PE)를 격자 형태로 배열**하고, 데이터가 격자를 따라 흘러가면서 각 PE가 연산을 수행하도록 설계했습니다.

### 이름의 유래

"Systolic(시스톨릭)"은 심장의 **수축기(systole)** 에서 따온 이름입니다. 심장이 박동할 때마다 혈액이 동맥을 따라 흘러가듯이, 시스톨릭 어레이에서는 **클럭 사이클마다 데이터가 PE에서 PE로 규칙적으로 흘러갑니다.**

```
클럭 1:  [데이터 A] → [PE₀₀] → [PE₀₁]
클럭 2:             [PE₀₀] → [PE₀₁] → (출력)
                       ↓         ↓
클럭 1:           [PE₁₀] → [PE₁₁]
클럭 2:                    [PE₁₀] → [PE₁₁] → (출력)
```

## 2. PE(Processing Element) 구조와 MAC 연산

### PE의 정의

PE는 시스톨릭 어레이를 구성하는 최소 연산 단위입니다. 각 PE는 매우 단순한 역할만 수행합니다.

1. **왼쪽 또는 위**에서 데이터를 수신합니다.
2. **MAC(Multiply-Accumulate)** 연산을 수행합니다.
3. 결과 또는 원본 데이터를 **오른쪽 또는 아래**로 전달합니다.

MAC 연산은 다음과 같이 정의됩니다.

$$\text{output} \mathrel{+}= A \times B$$

### PE 내부 구조 (단순화)

```
        [A 입력 (왼쪽)]
              │
              ▼
  ┌───────────────────────┐
  │   PE (i, j)           │
  │                       │
  │  acc += a_in × b_in   │──→ [A 출력 (오른쪽)]
  │                       │
  └───────────────────────┘
              │
              ▼
        [B 출력 (아래)]
        (또는 누산값 보관)
```

- `a_in`: 왼쪽 이웃으로부터 받은 행렬 $A$의 원소
- `b_in`: 위쪽 이웃으로부터 받은 행렬 $B$의 원소
- `acc`: 지역 누산 레지스터 (행렬 곱 결과 $C$의 해당 원소)

### SAVANT PE 코드 예시

```cpp
// 각 아두이노 보드(PE)의 핵심 연산
int acc = 0;  // 지역 누산기

void performMAC(int a_in, int b_in) {
  acc += a_in * b_in;  // MAC 연산
  // a_in은 오른쪽 PE로 전달, b_in은 아래 PE로 전달
  sendRight(a_in);
  sendDown(b_in);
}
```

<Note>
각 PE는 연산 중 **메모리에 직접 접근하지 않습니다.** 모든 데이터는 이웃 PE로부터 수신하고, 연산 후 다음 PE로 전달됩니다. 이것이 시스톨릭 어레이가 폰 노이만 병목을 우회하는 핵심 메커니즘입니다.
</Note>

## 3. 데이터 재사용 원리

### 재사용의 기본 개념

폰 노이만 방식에서는 행렬 원소가 필요할 때마다 메모리에서 꺼내야 합니다. $2 \times 2$ 행렬 곱의 경우 $A$와 $B$의 원소를 각각 **여러 번** 반복하여 읽어야 합니다.

반면 시스톨릭 어레이에서는 데이터가 격자의 한쪽 끝에서 입력되어 흘러가면서 **여러 PE가 동일한 데이터를 재사용합니다.**

### 2×2 격자에서의 데이터 흐름 예시 (OS 방식)

행렬 $A$의 원소는 **좌 → 우**로, 행렬 $B$의 원소는 **위 → 아래**로 흐릅니다.

```
시각 t=1:                 시각 t=2:
  A[0][0]→ [PE₀₀] [PE₀₁]    A[0][1]→ [PE₀₀]←A[0][0] [PE₀₁]
  B[0][0]↓              ↓    B[0][1]↓          B[0][0]↓
           [PE₁₀] [PE₁₁]              [PE₁₀]  [PE₁₁]
  A[1][0]→              ↓              ↓
  B[1][0]↓
```

격자에 한 번 입력된 $A[0][0]$은 **PE₀₀ → PE₀₁**을 거치며 두 번의 MAC 연산에 기여합니다. 메모리에서는 단 한 번만 읽혔지만, 두 PE가 이를 재사용합니다.

### 폰 노이만 vs 시스톨릭: 접근 횟수 비교

$M \times N$ 행렬 $A$와 $N \times P$ 행렬 $B$의 곱에서

| 방식 | 총 메모리 읽기 횟수 | 스케일링 |
|------|---------------------|----------|
| 폰 노이만 순차 | $M \times N \times P$ | 세제곱 |
| 시스톨릭 어레이 | $M \times N + N \times P$ (입력 1회) | 제곱 |

$M = N = P = 16$일 때: 폰 노이만 4,096회 vs 시스톨릭 512회(약 **8배 감소**).

## 4. 구글 TPU 적용 사례

### TPU의 탄생 배경

구글은 2015년 딥러닝 추론 워크로드가 급증하면서 기존 CPU·GPU로는 전력 효율과 처리 속도를 동시에 만족할 수 없다는 한계에 봉착했습니다. 이를 해결하기 위해 설계된 것이 **TPU(Tensor Processing Unit)** 이며, 핵심 연산 엔진으로 시스톨릭 어레이를 채택했습니다.

### TPU 시스톨릭 어레이의 규모

- 1세대 TPU: **256 × 256 = 65,536개**의 MAC 유닛이 시스톨릭 격자 형성
- 각 클럭 사이클(700 MHz)마다 전체 PE가 동시에 MAC 수행
- 이론적 최대 성능: $65{,}536 \times 700 \times 10^6 \times 2 \approx 92\ \text{TOPS}$ (INT8 기준)

### TPU가 같은 칩에 PE를 집적하는 이유

TPU에서 PE-PE 간 통신은 **같은 실리콘 위의 금속 배선**을 통해 이루어집니다. 이 때 신호 지연은 나노초(ns, $10^{-9}$s) 단위입니다.

<Warning>
**SAVANT 실험과 TPU의 결정적 차이: PE-PE 통신 지연**

SAVANT의 보드-보드 간 통신은 SoftwareSerial을 사용하며, 바이트당 약 **1 ms($10^{-3}$s)**의 전송 시간이 소요됩니다. 이는 TPU 내부 PE-PE 지연보다 **약 100만 배** 느립니다. SAVANT는 시스톨릭 어레이의 *알고리즘적 구조*(데이터 재사용, 병렬 MAC)를 재현하지만, 실제 칩 수준의 속도 이점은 통신 오버헤드로 인해 감쇄됩니다. 이 한계를 보고서에서 명시해야 합니다.
</Warning>

| 비교 항목 | SAVANT (보드 간) | TPU (칩 내부) |
|-----------|-----------------|---------------|
| PE-PE 지연 | ~1 ms/byte | ~1 ns |
| PE 수 | 4 (2×2) | 65,536+ |
| 통신 매체 | SoftwareSerial 선 | 금속 배선 |
| 주파수 | ~16 MHz | ~700 MHz |

## 5. 시스톨릭 어레이의 한계

### 타이밍 스큐(Timing Skew) 요구

시스톨릭 어레이가 올바르게 동작하려면 행렬 원소가 **정확한 클럭 사이클에 격자의 올바른 PE로 진입**해야 합니다. 이를 위해 데이터 입력 시 의도적인 지연(skew)을 삽입합니다.

$$\text{PE}_{(i,j)}\text{에 데이터 도달 시각} = t_{\text{start}} + i + j$$

격자가 커질수록 스큐 관리가 복잡해지며, 타이밍이 어긋나면 잘못된 원소끼리 곱해져 결과가 손상됩니다.

### 규칙적 연산에 특화

시스톨릭 어레이는 **행렬 곱셈처럼 규칙적이고 반복적인 연산**에 최적화되어 있습니다. 분기(branch)가 많거나 메모리 접근 패턴이 불규칙한 범용 코드에는 적합하지 않습니다.

| 적합한 워크로드 | 부적합한 워크로드 |
|----------------|-----------------|
| 행렬 곱셈, 합성곱 | 재귀, 포인터 추적 |
| 신경망 순전파/역전파 | 운영체제 커널 |
| 이미지 필터링 | 데이터베이스 조인 |

## 6. SAVANT 실험에서의 구현

### 하드웨어 구성

```
        [마스터 보드 - PE₀₀]──SoftwareSerial──[PE₀₁]
               │                                  │
           I²C/Serial                         Serial
               │                                  │
           [PE₁₀]──────SoftwareSerial──────[PE₁₁]
```

- **PE₀₀ (마스터)**: 입력 행렬 수신, 클럭 신호 생성, 최종 결과 수집
- **PE₀₁, PE₁₀**: 중간 처리 및 데이터 전달
- **PE₁₁**: 우하단 출력 원소($C[1][1]$) 담당

### 단순화된 동기 모델

실제 시스톨릭 어레이는 타이밍 스큐로 데이터 입력을 조절하지만, SAVANT는 **마스터 보드가 클럭을 발행하는 단순 동기 모델**로 구현합니다.

```cpp
// 마스터(PE₀₀)의 클럭 발행 예시
void clockPulse() {
  digitalWrite(CLK_PIN, HIGH);
  delayMicroseconds(50);
  digitalWrite(CLK_PIN, LOW);
  delayMicroseconds(50);
}

// 슬레이브 PE의 클럭 감지 후 데이터 수신
void onClockRise() {
  int received = receiveByte();  // SoftwareSerial로 수신
  acc += received * localWeight; // MAC 연산
}
```

<Tip>
타이밍 스큐 없이도 $2 \times 2$ 행렬에서는 올바른 결과를 얻을 수 있습니다. 그러나 격자가 $3 \times 3$ 이상으로 커지면 스큐 관리가 필수적입니다. 이를 향후 연구 방향으로 제시할 수 있습니다.
</Tip>

### 구현 요약

| 항목 | 내용 |
|------|------|
| PE 수 | 4개 (2×2 격자) |
| PE-PE 통신 | SoftwareSerial (커스텀 클럭 동기화) |
| MAC 연산 | 각 보드 내 `acc += a * b` |
| 타이밍 모델 | 단순 동기 (스큐 미적용, 한계로 명시) |
| 지원 데이터 흐름 | OS(출력 고정형), WS(가중치 고정형) |
