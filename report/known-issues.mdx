SAVANT 프로젝트의 실험 설계 및 구현 과정에서 총 10가지 설계 이슈가 발견되었다. 일부는 하드웨어 플랫폼의 구조적 한계에서 비롯된 것이고, 일부는 소프트웨어 구현 과정에서 발생한 타이밍 및 측정 기준의 문제다. 각 이슈는 보고서의 특정 섹션(한계점, 결과 분석, 실험 설계)과 직접 연결되므로, 이슈를 해결하는 방식과 보고서에 어떻게 기술해야 하는지를 함께 정리하였다. 이슈는 보고서 신뢰성에 미치는 영향도에 따라 **높음 / 중간 / 낮음** 세 단계로 분류되어 있다.

## 🔴 우선순위 높음 (3가지)

우선순위 높음 이슈는 방치하면 보고서의 핵심 논리 또는 실험 결과 자체에 영향을 미친다. 반드시 해결하고, 보고서 한계점 섹션에 명시적으로 기술해야 한다.

<Accordion title="이슈 #1 — Arduino 하버드 구조: 폰 노이만 병목 재현 불가">
  **우선순위:** 🔴 높음

  **문제 설명**

  SAVANT 실험의 대조군은 "폰 노이만 순차 처리 방식"을 구현하는 것을 목표로 한다. 그러나 Arduino Nano에 탑재된 ATmega328P는 순수 폰 노이만 구조가 아닌 **수정된 하버드 구조(Modified Harvard Architecture)** 를 사용한다. 수정된 하버드 구조에서는 프로그램 메모리(Flash)와 데이터 메모리(SRAM)가 물리적으로 분리되어 있어, 폰 노이만 구조의 핵심 특성인 "CPU와 메모리가 하나의 버스를 공유하는 병목"이 칩 수준에서는 발생하지 않는다. 따라서 대조군이 진정한 폰 노이만 병목을 재현하지 못한다는 한계가 있다.

  **해결 방법**

  대조군을 "폰 노이만 구조 자체"가 아닌 **"공유 배열(SRAM)에 순차적으로 접근하는 메모리 접근 패턴의 소프트웨어적 모사"** 로 명확히 재정의한다. 즉 실험 설계 단계에서부터 대조군의 역할을 "폰 노이만 병목을 하드웨어 수준에서 재현하는 것"이 아니라 "동일한 행렬 연산을 순차 방식으로 처리할 때의 메모리 접근 횟수를 소프트웨어적으로 측정하는 것"으로 정의한다.

  **보고서 기술 방법**

  이론적 배경 §2(하버드 구조와 아두이노) 및 실험의 한계점 섹션 두 곳 모두에 다음 내용을 명시해야 한다:

  > "본 실험에서 사용한 Arduino Nano(ATmega328P)는 수정된 하버드 구조를 채택하고 있으므로, 폰 노이만 구조의 메모리 병목을 하드웨어 수준에서 완전히 재현하지 못한다. 따라서 대조군 실험은 공유 SRAM 배열에 순차적으로 접근하는 패턴을 소프트웨어적으로 모사한 것으로 이해해야 하며, 이는 이 실험의 구조적 한계 중 하나이다."

  <Warning>
    이 이슈를 보고서에 기술하지 않으면, 심사자가 "아두이노는 폰 노이만 구조가 아닌데 왜 폰 노이만 대조군이라고 했는가"라는 치명적 지적을 할 수 있다.
  </Warning>
</Accordion>

<Accordion title="이슈 #2 — 통신 오버헤드: 병렬 처리 이득을 압도할 가능성">
  **우선순위:** 🔴 높음

  **문제 설명**

  시스톨릭 어레이 실험군에서 PE 간 데이터 전송은 아두이노 보드 간 외부 점퍼선을 통한 직렬 통신으로 이루어진다. SoftwareSerial 방식의 경우 9600baud 기준으로 약 1ms/byte의 전송 시간이 소요된다. 반면 SRAM 직접 접근 시간은 수 나노초(ns) 수준이다. 이로 인해 PE 간 통신 오버헤드가 병렬 처리로 얻는 연산 이득을 압도하여, 시스톨릭 어레이 방식이 오히려 더 느린 처리 시간을 기록할 가능성이 매우 높다.

  **해결 방법**

  두 가지 방향으로 대응한다:

  1. **기술적 최적화**: SoftwareSerial 대신 커스텀 클럭 동기 통신 방식을 구현하여 통신 속도를 높인다. 클럭 핀과 데이터 핀을 직접 제어하는 비트 단위 전송을 사용하면 SoftwareSerial 대비 오버헤드를 줄일 수 있다.
  2. **분석 전략 전환**: 결과가 예상(시스톨릭 > 순차)과 반대로 나오더라도, "느린 이유"에 대한 분석 자체를 실험의 결과물로 삼는다. 외부 보드 간 통신 오버헤드가 병렬 처리 이득을 상쇄한다는 것은, 역으로 "TPU가 왜 온칩(on-chip) PE 배선을 사용하는가"를 실험적으로 역증명하는 근거가 된다.

  **보고서 기술 방법**

  결과 분석 섹션에서 다음을 명시한다:

  > "실험에서 측정된 시스톨릭 어레이의 처리 시간이 순차 처리보다 길게 나타난 경우, 이는 PE 간 외부 클럭 동기 통신(수십~수백 µs/전송)이 SRAM 직접 접근(수 ns)보다 수천 배 느리기 때문이다. 이 결과는 TPU처럼 동일 칩 내부 배선(나노초 이하)을 사용할 때만 시스톨릭 어레이의 통신 비용이 무시할 수 있는 수준이 된다는 점을 역으로 보여준다."

  <Warning>
    이 이슈를 분석하지 않고 결과만 제시하면 "왜 느린가"를 설명하지 못한 보고서가 된다. 결과가 어떻게 나오든 반드시 통신 오버헤드를 분석 대상으로 삼아야 한다.
  </Warning>
</Accordion>

<Accordion title="이슈 #3 — PE(1,1) 동시 수신 문제: 두 소스 신호 누락">
  **우선순위:** 🔴 높음

  **문제 설명**

  2×2 시스톨릭 어레이에서 PE(1,1)은 PE(0,1)로부터 A 값을, PE(1,0)으로부터 B 값을 거의 동시에 수신해야 한다. 단순한 순차적 수신 코드(먼저 A를 기다린 뒤 B를 기다리는 방식)를 사용하면, 두 신호가 동시에 도달했을 때 먼저 온 신호를 처리하는 동안 나중 신호를 놓칠 수 있다.

  **해결 방법**

  두 입력 핀을 번갈아가며 폴링(polling)하는 루프 구조를 사용한다. 구체적으로는 `bool a_ready`, `bool b_ready` 플래그를 두고, 어느 쪽 클럭 핀이 HIGH가 되더라도 해당 데이터를 즉시 읽고 플래그를 세운 뒤, 두 플래그가 모두 세워졌을 때 MAC 연산을 실행한다.

  ```cpp
  // PE(1,1) 수신 루프 예시 패턴
  bool a_ready = false, b_ready = false;
  int8_t a_val, b_val;

  while (!a_ready || !b_ready) {
    if (!a_ready && digitalRead(CLK_A_PIN) == HIGH) {
      a_val = readBitsOnly(DATA_A_PIN, CLK_A_PIN);
      a_ready = true;
    }
    if (!b_ready && digitalRead(CLK_B_PIN) == HIGH) {
      b_val = readBitsOnly(DATA_B_PIN, CLK_B_PIN);
      b_ready = true;
    }
  }
  localSum += a_val * b_val;
  ```

  **보고서 기술 방법**

  실험 설계 섹션의 PE(1,1) 구현 설명 부분에서 이 폴링 방식을 명시한다. 한계점에는 별도 기술 불필요. 오히려 "동시 수신 문제를 폴링 방식으로 해결하였다"고 서술하면 설계의 완성도를 보여줄 수 있다.
</Accordion>

## 🟡 우선순위 중간 (4가지)

우선순위 중간 이슈는 구현 정확도와 측정 공정성에 영향을 미친다. 해결하지 않으면 실험 결과 해석에 문제가 생기므로 반드시 처리하되, 보고서 한계점에 간략히 언급하는 수준으로 기술한다.

<Accordion title="이슈 #4 — receiveByte 클럭 감지 충돌: 첫 비트 누락">
  **우선순위:** 🟡 중간

  **문제 설명**

  폴링 루프에서 클럭 핀이 HIGH임을 감지한 뒤 `receiveByte()` 함수를 호출할 때, `receiveByte()` 내부에서도 클럭 HIGH를 기다리는 코드가 있으면 문제가 발생한다. 폴링 루프에서 이미 HIGH 상태를 감지하고 함수를 호출했는데, 함수 내부에서 다시 "클럭이 HIGH가 되기를 기다린다"면 이미 전송이 진행 중인 상태가 되어 첫 번째 비트가 누락된다.

  **해결 방법**

  `receiveByte()` 함수를 두 가지 역할로 분리한다:
  - **클럭 감지**: 폴링 루프에서만 담당 (외부에서 처리)
  - **비트 읽기**: `receiveByte()` 내부에서는 클럭을 기다리지 않고, 클럭 타이밍에 맞춰 비트만 읽는 구조로 변경

  즉 폴링 루프가 "문이 열렸다"는 것을 감지하면, `receiveByte()`는 그 이후부터 비트를 순서대로 읽기만 하면 된다.

  **보고서 기술 방법**

  구현 세부 사항(코드 설계) 섹션에서 "수신 함수는 클럭 감지와 비트 읽기를 분리한 구조로 설계하였다"고 한 문장으로 언급한다. 한계점 기술 불필요.
</Accordion>

<Accordion title="이슈 #5 — D6 측정 구간 정의: 불공정 비교 방지">
  **우선순위:** 🟡 중간

  **문제 설명**

  처리 시간을 측정하기 위해 PE(0,0)에서 D6 핀을 HIGH로 설정하는 시점이 중요하다. 만약 D6 HIGH 설정이 PE(0,0)이 데이터를 전송한 **직후**에 이루어지면, PE(0,0) 자신의 MAC 연산 시간이 측정 구간에서 빠져 불공정한 비교가 된다. 즉 대조군(순차 처리)은 전체 연산 시간을 측정하면서 실험군(시스톨릭 어레이)은 일부 시간을 빠뜨리는 상황이 발생한다.

  **해결 방법**

  D6 핀을 HIGH로 설정하는 시점을 PE(0,0)의 **MAC 연산 시작 직전**으로 변경한다. 이렇게 하면 측정 구간이 "첫 번째 PE가 첫 번째 곱셈을 시작하는 순간"부터 "마지막 결과가 완성되는 순간"까지로 일관성 있게 정의된다.

  ```
  [잘못된 순서]  전송 완료 → D6 HIGH → MAC 연산 → ...
  [올바른 순서]  D6 HIGH → MAC 연산 시작 → ... → D6 LOW
  ```

  **보고서 기술 방법**

  실험 방법 섹션의 "처리 시간 측정 방법" 항목에서 "D6 핀은 PE(0,0)의 MAC 연산 시작 직전에 HIGH로 설정하여 측정 구간의 일관성을 확보하였다"고 명시한다.
</Accordion>

<Accordion title="이슈 #6 — 'AI 에너지 수십 배' 표현: 근거 없는 수치 과장">
  **우선순위:** 🟡 중간

  **문제 설명**

  보고서 초안에서 "AI 연산은 일반 연산보다 수십 배 더 많은 에너지를 소비한다"는 식의 표현을 사용할 경우, 이를 뒷받침하는 구체적 수치와 출처가 필요하다. 만약 de Vries(2023)를 인용하면서 이 논문이 그런 수치를 직접 제시하지 않는다면, 잘못된 인용이 된다. 근거 없는 수치 주장은 보고서 전체의 신뢰성을 손상시킨다.

  **해결 방법**

  "수십 배"와 같은 구체적 배수 표현 대신, de Vries(2023)가 실제로 제시하는 내용(AI 서비스 확산에 따른 데이터센터 전력 수요 증가 추세)을 인용한다. 폰 노이만 병목의 역할은 "유일한 원인"이 아닌 "구조적 비효율을 심화시키는 요인"으로 표현을 완화한다.

  **보고서 기술 방법**

  탐구 동기 문단에서 다음 표현 원칙을 따른다:
  - ❌ "AI 연산은 수십 배 더 많은 에너지를 소비한다"
  - ✅ "AI 서비스 확산으로 데이터센터 전력 수요가 폭발적으로 증가하고 있다(de Vries, 2023)"
  - ❌ "폰 노이만 구조가 AI 에너지 소비의 근본 원인이다"
  - ✅ "폰 노이만 구조의 구조적 비효율이 에너지 소비를 심화시키는 요인으로 작용한다"

  <Warning>
    인용한 논문이 실제로 주장하지 않는 수치를 해당 논문을 인용하여 뒷받침하는 것은 학문적 인용 오용(citation misuse)이다. 심사자가 원문을 확인할 경우 즉각 발각된다.
  </Warning>
</Accordion>

<Accordion title="이슈 #7 — '근본 원인' 논리적 비약: 복합 원인 무시">
  **우선순위:** 🟡 중간

  **문제 설명**

  "폰 노이만 병목이 AI 에너지 소비의 근본 원인"이라는 표현은 논리적 비약이다. AI 에너지 소비에는 ① 모델 파라미터 수(수천억 개), ② 학습 반복 횟수, ③ 데이터센터 냉각 등 인프라 비용, ④ 폰 노이만 구조의 메모리 비효율이 복합적으로 작용한다. 폰 노이만 병목만을 "근본 원인"이라 특정하면 나머지 요인들을 무시하는 것이 된다.

  **해결 방법**

  표현을 "구조적 비효율"로 조정하고, AI 에너지 소비의 원인이 복합적임을 인정하는 문장을 추가한다. 즉 "연산량 자체의 증가와 더불어, 폰 노이만 구조의 구조적 비효율이 함께 작용하고 있다"는 식의 병렬 구조 표현이 정확하다. (완성된 탐구 동기 문단이 이미 이 방식으로 작성되어 있다.)

  **보고서 기술 방법**

  탐구 동기, 결론 모두에서 "폰 노이만 구조가 AI 에너지 소비의 유일한 원인"이 아님을 명시한다. 결론에서 "본 실험은 여러 원인 중 구조적 비효율 측면에 집중하여 분석하였다"는 범위 한정 표현을 추가하면 논리적 완결성이 높아진다.
</Accordion>

## 🟢 우선순위 낮음 (3가지)

우선순위 낮음 이슈는 실험의 완전성이나 재현성과 관련된 사항이다. 결과의 핵심 논리에는 영향을 미치지 않으나, 보고서 한계점 또는 추후 과제 섹션에 간략히 언급하면 보고서의 완성도가 높아진다.

<Accordion title="이슈 #8 — WS 모드 결과 검증: PE(0,1) 출력 확인 방법">
  **우선순위:** 🟢 낮음

  **문제 설명**

  WS 모드로 행렬곱 C = A×B를 계산할 때, C 행렬의 일부 원소(예: C[0][p])는 PE(0,1)이 계산한다. 그런데 PE(0,1)의 출력이 LCD 화면에 표시되지 않도록 설계되어 있다면, 해당 값이 올바르게 계산되었는지 육안으로 확인할 수 없다.

  **해결 방법**

  코드 업로드 및 검증 단계에서 PE(0,1) 역할을 맡은 아두이노 보드를 노트북에 USB로 연결하고, Arduino IDE의 시리얼 모니터를 열어 `Serial.println(localSum)` 출력값을 확인한다. 검증이 완료된 뒤 노트북을 분리하고 정규 실험을 진행하면 된다.

  **보고서 기술 방법**

  실험 검증 방법 항목에서 "PE(0,1)의 출력은 코드 업로드 단계에서 시리얼 모니터로 별도 확인하였다"고 한 문장으로 언급한다. 한계점 기술 불필요.
</Accordion>

<Accordion title="이슈 #9 — 인위적 지연 추가: 결과 조작 오해 가능성">
  **우선순위:** 🟢 낮음

  **문제 설명**

  대조군(순차 처리)이 폰 노이만 병목을 더 실감나게 재현하기 위해 DRAM 접근 지연을 모사하는 `delay()` 함수를 추가하는 방안이 검토되었다. 그러나 이는 실험자가 결과를 원하는 방향으로 유도하기 위해 인위적으로 지연을 추가하는 것으로 비칠 수 있어, 실험의 공정성과 신뢰성을 심각하게 훼손한다.

  **해결 방법**

  메인 실험에서는 인위적 `delay()` 추가를 완전히 배제한다. 대조군은 아두이노 SRAM에 실제로 접근하는 시간 그대로 측정한다. 다만 "DRAM 접근 지연이 존재하는 실제 컴퓨터 환경에서는 어떤 차이가 날 것인가"라는 추가 논의(Discussion) 섹션을 설치하여, 이 시나리오를 이론적 추론으로만 다루는 것은 허용된다.

  **보고서 기술 방법**

  한계점 또는 추가 논의 섹션에서 다음과 같이 언급할 수 있다:

  > "실제 DRAM 접근 지연(수십~수백 나노초)을 모사하기 위한 인위적 지연 추가는 실험의 공정성을 해칠 수 있어 메인 실험에서 제외하였다. 이 요소가 포함될 경우의 영향은 추가 논의 섹션에서 이론적으로만 분석한다."
</Accordion>

<Accordion title="이슈 #10 — 타이밍 스큐 미적용: 단순화된 동기식 모델">
  **우선순위:** 🟢 낮음

  **문제 설명**

  실제 시스톨릭 어레이에서는 데이터를 배열 가장자리에서 입력할 때 각 행(또는 열)에 한 클럭씩 시간차(스큐, skew)를 두고 흘려보낸다. 이 스큐 덕분에 모든 PE가 정확히 올바른 타이밍에 올바른 데이터를 받을 수 있다. SAVANT는 4개 아두이노 보드를 사용하는 단순화된 구조이므로, 이 스큐를 완전히 구현하지 않고 동기식 단순화 모델을 사용한다.

  **해결 방법**

  SAVANT의 2×2 구현에서는 데이터 흐름이 단순하여 스큐 없이도 올바른 연산이 가능하도록 설계한다(각 PE가 자신이 받아야 할 데이터를 명시적으로 받는 방식). 그러나 이 단순화가 의미하는 바를 보고서에 명확히 기술해야 한다.

  **보고서 기술 방법**

  한계점 섹션에 다음 내용을 명시한다:

  > "실제 시스톨릭 어레이 구현에서는 데이터를 배열 가장자리에 입력할 때 행별로 한 클럭씩 시간차(스큐)를 두어야 한다. 본 실험에서는 2×2 규모의 단순화된 동기식 모델을 사용하였으며, 배열 규모가 커질수록 스큐 관리가 구현의 핵심 과제가 된다는 점을 향후 확장 과제로 남긴다."

  <Note>
    스큐 미적용이 현재 2×2 실험 결과의 정확성에 영향을 미치지는 않지만, 이 사실을 언급하는 것이 실험자가 실제 시스톨릭 어레이 동작 원리를 정확히 이해하고 있음을 보여준다.
  </Note>
</Accordion>
