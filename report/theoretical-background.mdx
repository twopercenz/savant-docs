이론적 배경은 실험 설계의 근거를 독자에게 설명하는 섹션이다. 단순히 교과서 내용을 나열하는 것이 아니라, 각 이론이 SAVANT 실험의 어느 부분과 연결되는지를 명확히 보여주어야 한다. SAVANT에서 다루는 이론적 배경은 ① 폰 노이만 구조, ② 하버드 구조와 아두이노, ③ 시스톨릭 어레이, ④ 데이터 흐름 방식, ⑤ 행렬 연산, ⑥ 이론적 성능 모델(Nc 공식), ⑦ UART 통신의 7개 주제로 구성된다. 이 페이지에서는 각 섹션에 담아야 할 핵심 내용과 서술 전략을 안내하고, 완성된 예시문(시스톨릭 어레이 섹션)을 제공한다.

## 추천 섹션 구성 개요

| 섹션 번호 | 제목 | 실험 연결 포인트 |
|-----------|------|-----------------|
| 1 | 폰 노이만 구조 | 대조군(순차 처리) 구현 근거 |
| 2 | 하버드 구조와 아두이노 | 실험 플랫폼 선택의 한계 설명 |
| 3 | 시스톨릭 어레이 | 실험군(병렬 처리) 구현 근거 |
| 4 | 데이터 흐름 방식 (WS/IS/OS) | WS(가중치 고정) 모드 선택 이유 |
| 5 | 행렬 연산 | 실험 대상(2×2 행렬곱)의 수학적 배경 |

| 7 | UART·클럭 동기 통신 | PE 간 통신 방식 및 오버헤드 근거 |

## 섹션별 작성 가이드

### 섹션 1. 폰 노이만 구조

**담아야 할 핵심 내용:**
- 폰 노이만 구조의 정의: CPU, 메모리, I/O 장치가 단일 버스(공유 통로)로 연결된 구조
- 폰 노이만 병목(Von Neumann Bottleneck): CPU와 메모리가 하나의 통로를 공유하기 때문에 발생하는 데이터 전송 속도 한계
- 행렬곱 연산 시 같은 행·열 데이터를 반복 로드해야 하는 구조적 특성
- Backus(1978)의 "폰 노이만 스타일에서 프로그래밍을 해방시킬 수 있는가" 논문 언급 가능 (선택)

**서술 팁:**
- "버스"라는 용어를 처음 쓸 때 "(데이터가 오가는 공유 통로)"라고 괄호로 풀어쓸 것
- 폰 노이만 병목을 시각적으로 설명하려면 "CPU가 밥을 먹으려면 매번 냉장고(메모리)까지 걸어가야 한다" 같은 비유를 쓴 뒤 바로 정확한 기술 표현으로 전환할 것

<Note>
  보고서에서 "폰 노이만 구조"는 실험 대조군의 이론적 근거다. 즉 이 섹션은 "왜 대조군이 순차적으로 메모리에 접근하는 방식으로 설계되었는가"를 설명해야 한다.
</Note>

### 섹션 2. 하버드 구조와 아두이노

**담아야 할 핵심 내용:**
- 하버드 구조의 정의: 프로그램 메모리(Flash)와 데이터 메모리(SRAM)가 물리적으로 분리된 구조
- ATmega328P (Arduino Nano에 탑재)는 수정된 하버드 구조(Modified Harvard Architecture)를 사용함
- 수정된 하버드 구조에서는 일부 명령어를 통해 프로그램 메모리에서 데이터를 읽을 수 있어 순수 하버드와 다름
- SAVANT 대조군이 진짜 폰 노이만 병목을 재현하지 못하는 이유

<Warning>
  이 섹션은 SAVANT 실험의 중요한 한계를 설명하는 부분이다. "아두이노는 폰 노이만 구조다"라고 잘못 기술하면 보고서 전체의 신뢰성이 흔들린다. 반드시 "수정된 하버드 구조이므로, 본 실험의 대조군은 폰 노이만 구조의 메모리 접근 패턴을 소프트웨어적으로 모사한 것"이라고 명확히 서술해야 한다.
</Warning>

**서술 팁:**
- 하버드 구조와 폰 노이만 구조의 차이를 표로 정리하면 한눈에 비교하기 좋다
- 아두이노 Nano의 사양(ATmega328P, Flash 32KB, SRAM 2KB, 클럭 16MHz)을 명시하면 이후 실험 설계 섹션과 연결이 자연스러워진다

#### ▼ 섹션 3 완성 예시문 (그대로 사용 가능)

> **3.1 시스톨릭 어레이의 등장 배경**
>
> 1978년 카네기멜런 대학의 쿵(Kung)과 레이서슨(Leiserson)이 처음 제안한 시스톨릭 어레이는, 폰 노이만 구조의 메모리 병목 문제를 하드웨어 구조 차원에서 해결하기 위해 개발되었다. "시스톨릭(systolic)"이라는 명칭은 심장이 규칙적으로 박동하며 혈액을 전신에 보내는 것처럼, 클럭 신호가 한 번 들어올 때마다 데이터가 격자 안의 처리 장치 사이를 한 칸씩 규칙적으로 이동한다는 아이디어에서 따온 표현이다.
>
> **3.2 PE(Processing Element)의 구조와 MAC 연산**
>
> 시스톨릭 어레이는 PE(Processing Element)라 불리는 작은 연산 장치들이 격자(grid) 모양으로 배열된 구조이다. 각 PE는 매우 단순한 동작만 수행한다. 이웃 PE로부터 두 값을 받아 곱셈-누산(MAC, Multiply-Accumulate) 연산, 즉 localSum += A × B를 실행하고, 받은 값을 그대로 다음 PE로 전달한다. 핵심은 각 PE가 중앙 메모리에 직접 접근하지 않는다는 점이다. 데이터는 격자의 가장자리에서 한 번만 입력되고, 이후로는 PE 사이를 흘러다니면서 여러 번 재사용된다.
>
> **3.3 데이터 재사용 원리**
>
> 폰 노이만 구조에서 행렬곱을 계산하려면 같은 행·열의 값을 반복해서 메모리에서 읽어와야 한다. 반면 시스톨릭 어레이에서는 데이터를 한 번 메모리에서 가져온 뒤 PE 사이에서 직접 주고받으며 재사용하기 때문에, 메모리 접근 횟수 자체가 줄어든다. 이 데이터 재사용이 에너지 소비를 줄이고 처리 효율을 높이는 핵심 메커니즘이다.
>
> **3.4 구글 TPU의 적용 사례**
>
> 구글은 2016년부터 자사 딥러닝 워크로드를 위해 TPU(Tensor Processing Unit)를 설계했으며, 그 핵심 연산 장치로 시스톨릭 어레이를 채택했다. TPU 내부에는 수백×수백 개의 PE가 격자로 배열되어 있고, 신경망 추론과 학습에서 집중적으로 발생하는 행렬곱 연산을 일반 CPU나 GPU보다 훨씬 적은 메모리 접근 횟수로 처리할 수 있다. 이는 곧 더 적은 에너지로 더 빠른 연산이 가능함을 의미한다. 다만 TPU에서 PE 사이의 통신은 동일 칩 내부 배선 수준(나노초 이하)으로 이루어지는 반면, 본 실험에서는 아두이노 보드 간 외부 점퍼선을 통한 클럭 동기 통신(수십~수백 마이크로초)을 사용하므로, 이 통신 오버헤드가 실제 TPU와 본 실험 결과의 가장 큰 차이를 만들어낸다.

### 섹션 4. 데이터 흐름 방식 (WS/IS/OS)

**담아야 할 핵심 내용:**
- 시스톨릭 어레이에서 행렬곱 A×B를 계산할 때 어느 데이터를 PE에 고정하고 어느 데이터를 흘려보내는지에 따라 세 가지 모드가 있음
  - **WS (Weight Stationary)**: 가중치 행렬(A)을 PE에 고정, 입력(B)을 흘림
  - **IS (Input Stationary)**: 입력(B)을 PE에 고정, 가중치(A)를 흘림
  - **OS (Output Stationary)**: 누산 결과(C)를 PE에 고정, A와 B 모두 흘림
- SAVANT에서 WS 모드를 선택한 이유: 아두이노 간 데이터 전송을 최소화할 수 있어 통신 오버헤드를 줄이는 데 유리함
- 각 모드의 장단점 비교 (에너지 효율 관점)

**서술 팁:**
- WS/IS/OS를 처음 소개할 때 표로 정리한 뒤 본문에서 WS만 깊이 설명하는 방식이 효율적이다
- "본 실험에서는 WS 모드를 채택하였다"고 명시하고 이유를 한 문장으로 설명할 것

### 섹션 5. 행렬 연산

**담아야 할 핵심 내용:**
- 행렬곱(Matrix Multiplication)의 정의: C = A × B에서 C[i][j] = Σ A[i][k] × B[k][j]
- 2×2 행렬곱의 연산 횟수: 곱셈 8회, 덧셈 4회 (총 12회 산술 연산)
- 폰 노이만 구조에서 2×2 행렬곱 시 메모리 접근 횟수 계산
- 시스톨릭 어레이 WS 모드에서 동일 연산의 메모리 접근 횟수 계산
- 두 방식의 이론적 차이를 수치로 비교

**서술 팁:**
- 수식은 LaTeX 없이도 "C[0][0] = A[0][0]×B[0][0] + A[0][1]×B[1][0]" 형태로 풀어 쓸 수 있다
- 2×2로 한정한 이유(아두이노 핀 수와 SRAM 제약)를 이 섹션 말미에 언급하면 실험 설계와 자연스럽게 연결된다

### 섹션 6. 이론적 성능 모델 (Nc 공식)

**담아야 할 핵심 내용:**
- 시스톨릭 어레이 n×n에서 n×n 행렬곱 처리 시 클럭 사이클 수(Nc)의 이론 공식
- 일반식: `Nc = 4n - 2` (n=2인 경우 Nc = 6)
- 이 공식이 의미하는 바: 배열 크기가 커질수록 처리 효율이 높아지는 이유
- 본 실험(2×2)에서 예측 클럭 수와 실측값 비교에 활용하는 방법

<Warning>
  Nc 공식의 출처를 명확히 표기해야 한다. 이 공식은 Kung & Leiserson(1978) 원 논문 또는 이를 인용한 후속 문헌에서 유도된다. 보고서에 수식을 쓸 때는 반드시 "(Kung & Leiserson, 1978)" 또는 사용한 교재·문헌을 인용할 것. 수식만 쓰고 출처를 생략하면 표절 의혹이 생길 수 있다.
</Warning>

**서술 팁:**
- 공식을 제시한 뒤 n=2를 대입하여 Nc=6임을 직접 계산해 보이면 독자가 이해하기 쉽다
- "배열이 커질수록 효율이 높아진다"는 점을 표나 그래프로 n=2, 4, 8일 때 Nc 값을 나열하면 설득력이 강해진다

### 섹션 7. UART·클럭 동기 통신

**담아야 할 핵심 내용:**
- UART(Universal Asynchronous Receiver-Transmitter)의 기본 원리
- SoftwareSerial 방식의 한계: 9600baud ≈ 약 1ms/byte, SRAM 접근(수 나노초)에 비해 수천 배 느림
- SAVANT에서 커스텀 클럭 동기 통신을 사용하는 이유: 통신 오버헤드 최소화
- 클럭 동기 통신의 원리: 클럭 핀 HIGH/LOW 신호를 기준으로 데이터 비트를 한 비트씩 전송
- PE 간 통신 오버헤드가 실험 결과에 미치는 영향 (Known Issues §2와 연결)

**서술 팁:**
- 이 섹션의 목적은 "왜 SAVANT의 시스톨릭 어레이가 TPU보다 훨씬 느릴 수 있는가"를 이론적으로 설명하는 것이다
- 이 섹션에서 통신 오버헤드를 미리 설명해두면, 결과 분석 섹션에서 "시스톨릭 어레이가 예상보다 느린 이유"를 설명할 때 논리적 근거가 된다

## 이론적 배경 전체 서술 원칙

<Steps>
  <Step title="각 섹션 첫 문장에서 실험과의 연결 명시">
    "이 구조가 SAVANT 실험의 대조군 설계에서 어떤 역할을 하는가"를 독자가 항상 인식할 수 있도록 각 섹션 도입부에서 연결 맥락을 언급한다.
  </Step>
  <Step title="기술 용어 첫 등장 시 반드시 정의">
    MAC, PE, WS, UART, ATmega328P 등의 용어는 처음 나올 때 괄호 안에 풀어쓰거나 각주로 설명한다. 고등학생 보고서는 독자가 해당 분야 전문가가 아님을 전제한다.
  </Step>
  <Step title="수식·공식에는 반드시 출처 표기">
    Nc 공식, MAC 연산식, 메모리 접근 횟수 공식 등 외부 문헌에서 가져온 수식에는 (저자, 연도) 형태의 인용을 붙인다.
  </Step>
  <Step title="한계는 이론 섹션에서 선제적으로 언급">
    아두이노 하버드 구조 한계(섹션 2), 통신 오버헤드(섹션 7)처럼 실험의 한계를 이론 섹션에서 먼저 설명해두면 결과 분석에서 방어 논리가 자동으로 구성된다.
  </Step>
</Steps>
