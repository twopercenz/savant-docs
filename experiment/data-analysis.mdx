측정이 완료되면 9가지 조건 × 5회 반복의 데이터를 체계적으로 정리하고, 이론 클럭 수 공식과 비교하여 처리 구조·데이터 흐름·행렬 크기 각각의 영향을 분석합니다. 이 페이지에서는 스프레드시트 구조, 이론 클럭 수 계산 방법, 단일 보드 vs 격자·OS vs WS·이론값 vs 실측값의 세 가지 비교 분석 방법, 예상치 못한 결과의 해석 방법, 그래프 작성 권장안, 보고서 결론 문장 템플릿을 순서대로 설명합니다.

## 스프레드시트 구조

데이터는 하나의 시트에 아래와 같은 열 구조로 정리합니다.

| 열 | 항목 | 데이터 형식 |
|----|------|-----------|
| A | 조건 번호 | 1~9 |
| B | 처리 방식 | 단일 / 격자 |
| C | 데이터 흐름 | OS / WS / — |
| D | 행렬 크기 | 2×2×2 등 |
| E~I | 회차 1~5 처리 시간 (μs) | 정수 |
| J | 평균 처리 시간 (μs) | `=AVERAGE(E:I)` |
| K | 표준편차 (μs) | `=STDEV(E:I)` |
| L~P | 회차 1~5 접근 횟수 | 정수 |
| Q | 평균 접근 횟수 | `=AVERAGE(L:P)` |
| R | 측정 클럭 수 | `=J열×16` (자동 계산) |
| S | 이론 클럭 수 (Nc) | 수동 입력 |
| T | 오차율 (%) | `=(R-S)/S×100` |

<Note>
R열의 측정 클럭 수는 `평균 처리 시간(μs) × 16 MHz` 로 계산합니다. Arduino Nano의 클럭 속도가 16 MHz이므로 1 μs = 16 클럭 사이클입니다.
</Note>

## 이론 클럭 수(Nc) 계산

### 공식

$$Nc = 2 \times S_R + S_C + T - 2$$

### 차원 매핑

행렬 크기 M×N×P(A는 M×N, B는 N×P)에서 각 데이터 흐름 방식에 따라 변수를 다음과 같이 대응합니다.

| 데이터 흐름 | S_R | S_C | T |
|------------|-----|-----|---|
| WS | M | N | P |
| IS | N | P | M |
| OS | M | P | N |

### 3가지 행렬 크기별 계산 예시

**조건 M=2, N=2, P=2 (2×2×2)**

| 방식 | 계산 | Nc |
|------|------|----|
| WS | 2×2 + 2 + 2 − 2 | **6** |
| OS | 2×2 + 2 + 2 − 2 | **6** |

M=N=P=2인 경우 WS와 OS의 이론 클럭 수가 동일합니다.

**조건 M=2, N=2, P=4 (2×2×4)**

| 방식 | 계산 | Nc | 비고 |
|------|------|----|------|
| WS | 2×2 + 2 + 4 − 2 | **8** | |
| OS | S_C = P = 4 > 격자 열 수(2) | 타일링 필요 | P=2씩 2라운드 → Nc×2 |

OS 모드에서 S_C(=P)가 격자 열 수(2)를 초과하면 타일링을 적용합니다. P=4인 경우 P=2짜리 2라운드로 분할하므로 실효 Nc = 2 × (2×2 + 2 + 2 − 2) = **12**가 됩니다.

**조건 M=2, N=4, P=2 (2×4×2)**

| 방식 | 계산 | Nc |
|------|------|----|
| WS | 2×2 + 4 + 2 − 2 | **8** |
| OS | 2×2 + 2 + 4 − 2 | **8** |

<Note>
타일링이 필요한 조건(조건 5, OS 2×2×4)은 스프레드시트의 S열에 타일링 적용 Nc를 입력하고 비고란에 "타일링 2라운드"를 기재합니다.
</Note>

## 측정 시간을 클럭 수로 변환

```
측정 클럭 수 = 측정 처리 시간(μs) × 16
```

예시: 측정 처리 시간 = 875 μs
→ 측정 클럭 수 = 875 × 16 = **14,000 클럭**

스프레드시트 R열에 `=J2*16` 수식을 입력하면 자동으로 계산됩니다.

## 비교 분석 1: 단일 보드 vs 2×2 격자

같은 행렬 크기에서 단일 보드(조건 1~3)와 격자 OS·WS(조건 4~9)의 평균 처리 시간을 비교합니다.

| 비교 쌍 | 단일 조건 | 격자 OS 조건 | 격자 WS 조건 |
|--------|---------|------------|------------|
| 2×2×2 | 조건 1 | 조건 4 | 조건 7 |
| 2×2×4 | 조건 2 | 조건 5 | 조건 8 |
| 2×4×2 | 조건 3 | 조건 6 | 조건 9 |

**분석 질문:**
- 행렬 크기가 커질수록 격자의 처리 시간 증가율이 단일 보드보다 완만해지는가?
- 격자가 단일 보드보다 빨라지는 교차점(crossover)이 측정 범위 내에 존재하는가?
- 격자가 단일 보드보다 느린 경우, 그 차이(오버헤드)는 몇 μs인가?

**통신 오버헤드 정량화:**

```
통신 오버헤드 = 격자 처리 시간 − 단일 보드 처리 시간
```

격자가 더 느린 경우 이 값이 양수가 되며, PE 간 시리얼 통신에 소요되는 순수 오버헤드를 나타냅니다.

---

## 비교 분석 2: OS vs WS (격자 내부 비교)

같은 행렬 크기, 같은 격자 구조에서 OS 모드와 WS 모드를 비교합니다.

| 행렬 크기 | OS | WS | 비교 |
|----------|----|----|------|
| 2×2×2 | 조건 4 | 조건 7 | OS − WS |
| 2×2×4 | 조건 5 | 조건 8 | OS − WS |
| 2×4×2 | 조건 6 | 조건 9 | OS − WS |

**분석 질문:**
- WS 모드가 OS 모드보다 처리 시간이 짧은가? (가중치 전송 생략의 효과)
- 접근 횟수에서 두 모드 간 차이는 이론적으로 예측한 통신 감소량과 일치하는가?
- 타일링이 필요한 조건 5(OS, 2×2×4)에서 OS가 WS보다 현저히 느린가?

## 비교 분석 3: 이론 클럭 수 vs 측정 클럭 수

| 조건 | 이론 Nc | 측정 클럭 수 | 오차율(%) |
|------|--------|------------|---------|
| 조건 4 (OS, 2×2×2) | 6 | J열×16 | (측정−이론)/이론×100 |
| 조건 5 (OS, 2×2×4) | 12 (타일링) | J열×16 | |
| 조건 6 (OS, 2×4×2) | 8 | J열×16 | |
| 조건 7 (WS, 2×2×2) | 6 | J열×16 | |
| 조건 8 (WS, 2×2×4) | 8 | J열×16 | |
| 조건 9 (WS, 2×4×2) | 8 | J열×16 | |

**오차 원인 분석:**

이론 Nc는 순수 연산 클럭만 계산하며, 다음 요인들이 추가 클럭을 유발합니다.

| 오차 원인 | 설명 |
|---------|------|
| 시리얼 통신 지연 | SoftwareSerial 바이트 전송 시간 (9600 baud = 약 1 ms/바이트) |
| `parseInt()` 파싱 | 수신 버퍼 처리 오버헤드 |
| `delay()` 안정화 | 코드에 포함된 안정화 대기 시간 |
| 인터럽트 처리 | Arduino 내부 타이머 인터럽트 |
| D6 폴링 대기 | PE(1,1)의 D6 HIGH 대기 루프 시간 |

<Note>
이론값과 실측값의 비율(실측 / 이론)이 수십~수백 배에 달할 수 있습니다. 이는 이론 클럭 수 공식이 하드웨어 시스톨릭 어레이를 가정하는 반면, 실험은 소프트웨어 시리얼로 구현한 차이에서 비롯됩니다. 이 차이를 정량적으로 분석하는 것이 중요합니다.
</Note>

## 예상치 못한 결과 해석

### 격자가 단일 보드보다 느린 경우

<Warning>
격자가 단일 보드보다 느리게 측정되었다고 해서 실험이 실패한 것이 아닙니다. 소규모 행렬에서는 PE 간 통신 오버헤드가 병렬 처리 이득을 초과하는 것이 이론적으로도 예측 가능한 결과입니다.
</Warning>

분석 방향:
1. **오버헤드 정량화**: `격자 처리 시간 − 단일 보드 처리 시간 = 통신 오버헤드`를 계산합니다.
2. **행렬 크기와의 관계**: 행렬이 커질수록 오버헤드 비율이 감소하는 추세가 있는지 확인합니다.
3. **교차점 추정**: 현재 데이터에서 추세선을 그려 격자가 유리해지는 행렬 크기를 외삽(extrapolation)합니다. 측정 범위(최대 2×4×2)에서 교차가 발생하지 않더라도, 이론적 교차점을 추정하는 것 자체가 의미 있는 분석입니다.

```
오버헤드 비율(%) = (격자 시간 / 단일보드 시간 − 1) × 100
```

## 그래프 작성 권장안

### 그래프 1: 처리 시간 vs 행렬 크기 (선 그래프)

- X축: 행렬 크기 (2×2×2, 2×2×4, 2×4×2)
- Y축: 평균 처리 시간 (μs)
- 선 3개: 단일 보드 / 격자 OS / 격자 WS
- 오차 막대: 표준편차 (±1 SD)

단일 보드와 격자의 교차점이 시각적으로 드러납니다. 교차점이 측정 범위를 벗어나면 추세선(점선)을 연장하여 표시합니다.

### 그래프 2: 이론 클럭 수 vs 측정 클럭 수 (막대 그래프)

- X축: 조건 번호 (조건 4~9)
- Y축: 클럭 수 (log 스케일 권장 — 두 값의 차이가 클 경우)
- 막대 2개씩: 이론 Nc / 측정 클럭 수
- 각 막대 위에 오차율(%) 표기

이론과 실측의 차이 원인(통신 오버헤드)을 시각화합니다.

### 그래프 3: OS vs WS 처리 시간 비교 (묶음 막대 그래프)

- X축: 행렬 크기 3가지
- Y축: 평균 처리 시간 (μs)
- 막대 2개씩: OS / WS
- 오차 막대: 표준편차

WS 모드의 통신 감소 효과가 처리 시간에 미치는 영향을 시각화합니다.

## 보고서 결론 문장 템플릿

아래 문장들을 실제 측정값으로 채워 보고서 결론 섹션에 활용합니다.

**단일 보드 vs 격자 비교:**
> "2×2×2 행렬에서 단일 보드의 평균 처리 시간은 [___] μs, 2×2 격자(OS)는 [___] μs로, 격자가 단일 보드보다 [___]배 [빠르게/느리게] 처리하였다. 격자의 통신 오버헤드는 [___] μs로 추정된다."

**OS vs WS 비교:**
> "동일한 2×2×4 행렬에서 OS 모드의 평균 처리 시간은 [___] μs, WS 모드는 [___] μs였으며, WS 모드가 [___] μs([___]%) 더 [빠르게/느리게] 측정되었다. 이는 WS 모드에서 가중치 전송이 생략됨에 따른 통신 횟수 감소([___]회 차이)가 반영된 결과로 해석된다."

**이론값 vs 실측값 비교:**
> "이론 클럭 수 공식(Nc = 2×S_R + S_C + T − 2)에 따르면 WS 2×2×2 조건의 예상 클럭 수는 6이나, 실측값은 [___] 클럭으로 [___]배의 차이를 보였다. 이 차이는 SoftwareSerial 통신 지연([___] ms/바이트), parseInt() 파싱 오버헤드, Arduino 내부 인터럽트 등 소프트웨어 구현에 따른 추가 클럭에 기인한다."

**교차점 분석:**
> "현재 측정 범위(최대 2×4×2)에서 격자 구조가 단일 보드보다 빠른 교차점은 관찰되지 [않았다/조건 [___]에서 확인되었다]. 추세선 외삽 결과, 행렬 크기가 [약 M×N×P] 이상일 때 격자가 유리해질 것으로 추정된다."

<Tip>
결론 작성 시 "예상과 달리"보다 "측정 결과에 따르면"으로 시작하는 중립적 표현을 사용합니다. 예상과 다른 결과는 추가 논의(Discussion) 섹션에서 원인 분석과 함께 기술합니다.
</Tip>
